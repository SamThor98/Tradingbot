                  Onboarding logic
                  connect + verify sequence
                
                
      
                  
    • Connect account: links brokerage permissions used for orders and account context.
      • 
                  
    • Connect market data: links quote access used by scanning, charts, and checks.
      • 
                  
    • Verify tokens: confirms auth tokens are present and usable before live workflows.
      • 
                  
    • Test scan / paper order: validates end-to-end flow without risking live capital.
      • 
                
    
              
    
              
    
                
                  Preset control logic
                  how profile settings apply
                
                
      
                  
    • Profile: selects the risk/behavior template (conservative, balanced, aggressive).
      • 
                  
    • Mode: standard shows plain-language values; expert also shows raw runtime env values.
      • 
                  
    • Rank explain: controls how score rationale renders in scan tables (compact tooltip vs inline text).
      • 
                  
    • Automation opt-in: stores whether live automation can proceed without the extra confirmation gate.
      • 
                  
    • Apply preset: persists selected settings to backend profile config and reloads panel state.
      • 
                  
    • Open feature guide: manually reopens the initial website walkthrough popup.
      • 
                
    
              
    
              
    
                
                  Preset guardrails (trust + safety)
                  actual enforced values + why each exists
                
                
      
                  
    • Guardrails are fixed presets, not free-form: users can only apply conservative, balanced, or aggressive templates. Reason: removes accidental misconfiguration and keeps behavior auditable.
      • 
                  
    • Position size caps by preset: conservative $300, balanced $500, aggressive $900 per new buy. Reason: hard-limits single-trade blast radius by risk profile.
      • 
                  
    • Daily trade cap by preset: conservative 3, balanced 5, aggressive 8 new live trades/day. Reason: prevents over-trading and limits damage from bad market regimes.
      • 
                  
    • Quality-gate strictness: conservative uses QUALITY_GATES_MODE=hard; balanced/aggressive use soft. Reason: tighter profile rejects weaker setups; looser profiles allow more opportunity flow.
      • 
                  
    • Event-risk policy: conservative/balanced use EVENT_RISK_MODE=live; aggressive uses shadow. Conservative uses EVENT_ACTION=block, balanced/aggressive use downsize. Reason: earnings/macro risk is either blocked or size-reduced based on desired aggressiveness.
      • 
                  
    • Execution-quality policy: conservative/balanced use EXEC_QUALITY_MODE=live; aggressive uses shadow. Reason: stricter profiles enforce spread/slippage checks live; aggressive logs would-block signals without hard blocking.
      • 
                  
    • Automation defaults fail-safe: automation_opt_in defaults to off, so approvals require explicit live confirmation unless intentionally enabled. Reason: avoids silent automation on first use.
      • 
                  
    • Apply is explicit and reviewable: the UI shows a before/after preview, then persists profile + mode + automation state with an update timestamp. Reason: clear operator intent and change traceability.
      • 
                
    
              
    
              
    
                
                  What the screener looks for (trading rationale)
                  overall signal thesis in plain English
                
                
      
                  
    • Core edge thesis: the screener targets trend-continuation breakouts (Stage 2 + VCP behavior), not random short-term mean reversion. Reason: continuation setups historically provide cleaner follow-through when trend, structure, and participation align.
      • 
                  
    • Stage A (fast filter): quickly narrows the universe to candidates with Stage 2/VCP characteristics so capital is not spent evaluating weak names. Reason: speed + discipline — most symbols should fail fast.
      • 
                  
    • Stage B (enrichment): shortlisted names are stress-tested with deeper context (quality gates, event/forensic checks, SEC context, advisory probabilities, and strategy ensemble inputs). Reason: avoid false breakouts that look good on price alone.
      • 
                  
    • Regime awareness: market-regime gates can block or suppress risk when broad conditions are unfavorable. Reason: even good single-name setups underperform in weak tape, so the system is designed to reduce exposure then.
      • 
                  
    • Score = prioritization, not blind execution: score/edge/reliability/execution components rank opportunities by quality and tradeability. Reason: highest-ranked names should combine signal strength with practical execution quality, not just chart shape.
      • 
                  
    • Near-miss visibility: names that fail thresholds remain visible with reasons. Reason: transparency builds trust and helps users see that exclusions are rule-driven, not arbitrary.
      • 
                
    
              
    
              
    
                
                  Live trading logic
                  activation, pause, security, billing
                
                
      
                  
    • Enable live trading: requires risk acknowledgment + typing ENABLE before activation.
      • 
                  
    • Pause new live orders: blocks new approval execution until pause is removed and saved.
      • 
                  
    • 2FA panel: adds extra approval protection for sensitive/high-value actions.
      • 
                  
    • Billing controls: checkout starts subscription; portal manages payment method and plan lifecycle.
